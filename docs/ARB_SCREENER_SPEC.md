# Crypto Arbitrage Screener — спецификация реализации

Рабочий документ-источник правды для сервера, который скринит криптовалютные пары на
нескольких биржах, считает межбиржевые спреды и отдаёт сигналы клиентам. Цель на
перспективу — мультитенант SaaS с автоисполнением, но строим **фазами**, начиная с
безопасного скринера на публичных данных.

---

## 0. Правила для реализующего агента

Читать этот раздел перед тем, как писать код.

- **Фазность священна.** Не тащить исполнение и работу с торговыми ключами в Фазу 1.
  Каждая фаза должна быть самодостаточной и запускаемой.
- **Фаза 1 — только публичные данные, никаких ключей бирж.** Котировки/стаканы/фандинг
  доступны без авторизации. Ключи появляются только в фазе живого исполнения.
- **Публичные данные — общие на всех клиентов.** Одна связка подключений на биржу.
  Правила считаются per-client. Ключи и исполнение (позже) — строго изолированы per-tenant.
- **BBO не доверяем.** Везде, где считаем «выгоду», считаем *исполнимый* спред по глубине
  под целевой размер, а не только по лучшим ценам.
- **Наивный спред — мираж.** Всегда вычитаем комиссии тейкера с двух ног; фандинг и
  слиппедж учитываем там, где это применимо.
- **Свежесть данных — по локальному монотонному времени приёма**, не по таймстемпам бирж
  (рассинхрон часов). Устаревшие котировки помечаем/дропаем.
- **Секреты никогда не логируются.** `SecretString` + `zeroize`; при персисте — envelope-
  шифрование через KMS. В Фазе 1 секретов нет вовсе.
- **Реконнект обязателен и корректен:** экспоненциальный backoff с jitter, ресабскрайб,
  для L2-книг — ресинк из снапшота.
- **Латентность-реализм.** Быстрый межбиржевой арбитраж захвачен HFT с колокацией; для
  self-hosted SaaS целимся в латентность-толерантные стратегии (фандинг/базис/статарб).
  Быстрый spatial держим как источник алертов, не как автоисполнение.

### Конвенции кода

- Async на `tokio`. Ошибки: библиотечные крейты — `thiserror`, бинарь/склейка — `anyhow`.
- Никаких `unwrap()`/`expect()` на горячих путях; только в тестах и на заведомо-инвариантах.
- Всё конфигурируемо (биржи, символы, пороги) через `config` + env, не хардкод.
- Логи и трейсинг через `tracing` со спанами на подключение/инструмент.
- Числа цен/объёмов — `rust_decimal::Decimal` (не `f64`) во всех расчётах денег.

## 1. Контекст и цель

Сервер собирает рыночные данные с нескольких бирж, приводит их к единому виду, держит
актуальное состояние стаканов в памяти, вычисляет лучший межбиржевой спред по каждому
инструменту и по правилам клиента шлёт сигналы. В перспективе — исполнение сделок, но это
поздние фазы.

**Текущая фаза: 1 — скринер на публичных данных.**

## 2. Архитектура (слои и поток данных)

```
   Биржа A (WS/REST) ─┐
   Биржа B (WS/REST) ─┼─► [Connectors] ─► нормализация ─► MarketUpdate
   Биржа C (WS/REST) ─┘         │
                                ▼
                        [Ingest manager]  (реконнект, роутинг, health)
                                │
                                ▼
                        [Market State]    (по (exchange, instrument): topN стакана;
                                           инкрементальная сводка min-ask / max-bid)
                                │  событие обновления инструмента
                                ▼
                        [Screener engine]  (для каждого клиента: фильтры, исполнимый
                                            спред, комиссии, гистерезис, lifetime)
                                │  ScreenerEvent
                                ▼
                        [API / Delivery]   (axum: WS push клиентам, REST для конфига)
                                │
                                ▼
                           Клиенты (WS)

   Сквозные: [Auth клиентов] · [Observability] · [Persistence (опц.)]
   Поздние фазы: [Account State] · [Execution] · [Order Manager] · [Risk Manager]
                 · [Secrets/KMS] · [Reconciliation]
```

**Ключевое решение:** ingestion публичных данных общий на всех; изоляция per-tenant
начинается на уровне правил, а для ключей/исполнения — полная.

## 5. Доменная модель (`domain`)

Наброски типов (иллюстративно, не финально):

```rust
pub enum ExchangeId { Bybit, Binance, Okx /* ... */ }

pub enum MarketKind { Spot, Perp }

/// Канонический инструмент, единый для всех бирж.
pub struct Instrument {
    pub base: String,   // "BTC"
    pub quote: String,  // "USDT"
    pub kind: MarketKind,
}

pub struct BookLevel { pub price: Decimal, pub qty: Decimal }

/// Топ-N стакана по инструменту на конкретной бирже.
pub struct TopBook {
    pub bids: Vec<BookLevel>, // отсортированы по убыванию цены
    pub asks: Vec<BookLevel>, // по возрастанию цены
    pub recv_ts: Instant,     // локальное монотонное время приёма
    pub exch_ts: Option<i64>, // таймстемп биржи, только для справки
}

pub enum MarketUpdate {
    Book { exchange: ExchangeId, instrument: Instrument, book: TopBook },
    Funding { exchange: ExchangeId, instrument: Instrument, rate: Decimal, next_ts: i64 },
}

/// Итоговая лучшая связка по инструменту.
pub struct Spread {
    pub instrument: Instrument,
    pub buy_exchange: ExchangeId,   // где min ask
    pub sell_exchange: ExchangeId,  // где max bid
    pub gross_pct: Decimal,         // до комиссий
    pub net_pct: Decimal,           // после комиссий (и, где надо, фандинга)
    pub executable_notional: Decimal, // на какой размер (USDT) реально исполнимо
}
```

Трейт коннектора:

```rust
#[async_trait]
pub trait ExchangeConnector: Send + Sync {
    fn id(&self) -> ExchangeId;
    /// Подключиться, подписаться на symbols, гнать нормализованные апдейты в tx.
    /// Внутри: connect → subscribe → цикл чтения → нормализация → tx.send;
    /// + reconnect(backoff+jitter) + resubscribe + (для L2) ресинк книги.
    async fn run(
        self: Arc<Self>,
        symbols: Vec<Instrument>,
        tx: mpsc::Sender<MarketUpdate>,
    ) -> anyhow::Result<()>;
}
```

## 6. Нормализация

- **Символы.** У каждой биржи свой формат (`BTCUSDT`, `BTC-USDT`, `BTC/USDT`,
  `BTC-USDT-SWAP`). В коннекторе — двусторонний маппинг `Instrument <-> биржевой символ`.
  Внутри системы ходит только `Instrument`.
- **Единицы объёма** приводим к базовому активу (монете); нотионал считаем как `price * qty`.
- **Точность/тик-сайз** различаются — храним метаданные инструмента (min qty, min notional,
  tick size), понадобятся для исполнения; в Фазе 1 достаточно логировать.
- **Различия листингов.** Инструмент может быть не на всех биржах — это нормально; сводка
  считается по тем биржам, где он есть (нужно ≥2).

---

## 7. Состояние рынка (`market_state`)

- Хранилище: `(ExchangeId, Instrument) -> TopBook`. Конкурентный доступ (`dashmap` или
  шардированные локи).
- На каждом `MarketUpdate::Book` обновляем `TopBook` и запускаем пересчёт **сводки** по
  этому инструменту: перебор бирж, где он есть, — находим лучшую связку (см. §8).
- Устаревшие книги (`now - recv_ts > staleness_threshold`) исключаем из расчёта и помечаем
  биржу как «залипшую».
- Фандинг храним отдельно: `(ExchangeId, Instrument) -> (rate, next_ts)`.

## 8. Логика скрининга (ядро)

### 8.1 Лучшая связка и BBO-спред

Для инструмента `I` собираем по всем биржам валидные `(exchange, best_bid, best_ask)`
(условие валидности: `ask > 0 && bid > 0 && bid <= ask && !stale`). Перебором пар
`(buy=i, sell=j), i≠j, exch_i≠exch_j` ищем максимум `(bid_j - ask_i) / ask_i`. Число бирж
мало, полный перебор дёшев и корректно обрабатывает совпадения. Пары на одной бирже
пропускаем (это её внутренний спред, всегда отрицательный).

### 8.2 Исполнимый спред (главное отличие от наивного)

Под целевой размер `Q` (в quote-валюте, напр. 5000 USDT):

- **Покупка на бирже X:** идём по `asks` от лучшей вверх, накапливаем нотионал, пока не
  наберём `Q` (последний уровень — частично). `vwap_buy = потрачено / куплено_qty`.
- **Продажа на бирже Y:** идём по `bids` от лучшей вниз до `Q`. `vwap_sell`.
- `gross = (vwap_sell - vwap_buy) / vwap_buy`.
- `net = gross - fee_buy - fee_sell` (тейкер, доли; берём per-exchange из конфига).
- Если у обеих ног хватает глубины на `Q` **и** `net > threshold` — кандидат в сигнал.
- `executable_notional` = сколько реально удалось набрать (может быть < `Q`, если стакан
  тонкий) — сохраняем в сигнал.

> Если глубина стакана недоступна (только BBO), исполнимый спред считать нельзя — тогда
> сигнал помечается как «BBO-only, размер неизвестен». Для автоисполнения такие сигналы
> использовать нельзя.

### 8.3 Фандинг (для латентность-толерантных стратегий)

Для перпов держим текущий фандинг и время следующей выплаты. Считаем дифференциал фандинга
между биржами (annualized) как отдельный сигнал — это самый вероятный первый рабочий кейс
для SaaS. В Фазе 1 достаточно логировать; движок фандинг-арбитража — отдельное правило.

### 8.4 Гистерезис и дедуп (урок из OsEngine-бота)

На `(client, instrument)` храним «пик» уже отправленного нетто-спреда:

- нет пика и `net > threshold` → шлём сигнал, пишем `peak = net`;
- есть пик и `peak + step < net` → спред раздвинулся ещё → шлём, обновляем пик;
- `net <= threshold` → **сбрасываем** пик (удаляем запись), чтобы следующая раздвижка снова
  дала сигнал. (Без сброса получается «храповик», который замолкает после первого пика.)

### 8.5 Lifetime спреда (для выбора стратегии)

Когда `net` впервые пересекает порог — фиксируем `opened_at`; когда падает ниже — считаем
`duration = now - opened_at` и логируем. Распределение длительностей прямо показывает,
успеет ли self-hosted SaaS исполнять эти окна или это только алерты.

### 8.6 Фильтры клиента

`ClientConfig`: список бирж, allow/deny-лист инструментов (или маска), `min_net_spread`,
`min_executable_notional`, `target_size Q`, per-exchange комиссии. Движок применяет их
поверх общей сводки. Данные общие — расходятся только фильтры и состояние гистерезиса.

## 9. Доставка и API (`api`)

- **WS-хаб.** Клиент подключается, аутентифицируется, шлёт `Subscribe(ClientConfig)`,
  получает поток `ScreenerEvent { spread, executable_notional, ts }`.
- **Backpressure.** На клиента — bounded-канал + **коалесинг**: при заторе отправляем
  последнее состояние по инструменту, а не каждый тик. Медленный клиент не тормозит ядро.
- **REST.** Управление конфигом, разовые запросы текущей сводки, health/metrics.

## 10. Наблюдаемость и тестирование

- `tracing`-спаны на подключение/инструмент; структурные логи.
- Метрики: updates/sec на биржу, возраст данных, число реконнектов, размер очередей,

- **Мок-коннекторы** и **реплей записанных WS-фидов** — обязательны: реконнекты и сборку

- Unit-тесты на: нормализацию символов, VWAP/исполнимый спред, гистерезис (все ветки),
  дедуп, обработку stale.

## 11. Дорожная карта по фазам

**Фаза 1 — скринер (текущая).** Публичные данные, без ключей. Ingestion + нормализация +
market_state + screener + WS-выдача. Логируем BBO-спред, исполнимый спред по N уровням,
lifetime, фандинг. Цель — на реальных данных понять, что живёт, а что мираж.

**Фаза 2 — paper trading.** Учёт комиссий/фандинга/глубины и «бумажное» исполнение: считаем,
сколько бы заработали, без ордеров и без ключей.

**Фаза 3 — живое исполнение, 1 юзер, малый капитал.** Появляются `account_state` (нужны
ключи), `execution`, `order_manager`, `risk_manager`, `secrets/KMS`, `reconciliation`.
Капитал держится **заранее на обеих биржах** + периодический ребаланс (схема
«купить→перевести→продать» под автоарбитраж не работает — переводы слишком медленные).
Обрабатываем leg risk (одна нога исполнилась, вторая — нет), частичные филлы, реджекты,
идемпотентность, сверку.

**Фаза 4 — мультитенант.** Строгая изоляция тенантов (капитал, риск, учёт), честное
распределение, per-tenant kill-switch.

> Юридическое: торговля от имени пользователей с их ключами может подпадать под
> регулирование, а ToS ряда бирж ограничивают передачу ключей/стороннюю автоторговлю.
> Проверить с юристом до Фазы 3. (Не является юридической консультацией.)

## 12. Конкретные шаги реализации Фазы 1

1. Инициализировать workspace и крейт `domain` с типами и трейтом `ExchangeConnector`.
2. Реализовать **один** коннектор (по умолчанию **Bybit** или **Binance**): подписка на
   поток лучших цен (bookTicker) + частичную глубину (partial depth, N уровней) + фандинг;
   нормализация в `MarketUpdate`. Реконнект с backoff+jitter и ресабскрайбом.
3. `ingest::manager`: поднять коннектор(ы), собрать `mpsc` апдейтов, роутить в `market_state`.
4. `market_state`: хранилище `TopBook`, инкрементальная сводка min-ask/max-bid, отсечка stale.
5. `screener`: BBO-спред, исполнимый спред (VWAP по N уровням), вычет комиссий, гистерезис
   со сбросом, замер lifetime. Пока один «дефолтный клиент» из конфига.
6. Логирование сигналов + запись метрик; опционально дамп в CSV/БД для анализа lifetime.
7. Минимальная WS-выдача через `axum` (даже без аутентификации на старте — локально).
8. Записать реальный фид в файл, сделать реплей-тест нормализации и расчёта спреда.
9. Добавить вторую биржу по образцу первой — проверить кросс-биржевую сводку.

## 13. Открытые решения (уточнить по ходу)

- С какой биржи начинать (дефолт-предложение: **Bybit**).
- Глубина `N` уровней для исполнимого спреда (старт: 10–25).
- Целевой размер `Q` для расчёта исполнимости (задать в конфиге, напр. 1000–5000 USDT).
- Тип арбитража — **определяется данными Фазы 1**, не выбирается заранее.
- Нужна ли персистентность истории в Фазе 1 (для анализа lifetime — желательно CSV/ClickHouse).
